# Session: Auth — Database Schema
## Phase
1 — Auth & Identity

## Context files to load
- `SPEC.md` (section: Phase 1 — Database)
- `CLAUDE.md`
- `backend/shared/models/audit.py`

## Your task
Create the Alembic migration for `companies` and `user_profiles` tables exactly as specified in SPEC.md Phase 1 database section. Create the corresponding SQLAlchemy 2.0 async models in `backend/services/auth/models.py`. Both tables must include `created_at`. `user_profiles` must reference `companies(id)` via FK. Add the `deleted_at`, `created_by`, `updated_by` columns to both tables (financial record immutability rule). Add indexes on `user_profiles.keycloak_id` (UNIQUE), `user_profiles.company_id`, and `user_profiles.email`.

## Constraints
- Use `UUID PRIMARY KEY DEFAULT gen_random_uuid()` on both tables
- `user_profiles.keycloak_id` must be UNIQUE — one Keycloak identity maps to one profile
- `user_profiles.role` default: `'owner'`
- Both models inherit from a shared `Base` (SQLAlchemy declarative base in `backend/shared/`)
- All monetary amounts use `NUMERIC(12,2)` — none in this schema but establish the pattern
- Migration file must be reversible (include `downgrade()`)
- Never hard-code company_id in tests — use fixtures

## Done when
- `alembic upgrade head` applies migration cleanly from empty DB
- `alembic downgrade -1` reverses it cleanly
- `pytest backend/services/auth/tests/test_models.py -v` passes (create company, create user_profile, FK integrity)
- `make verify` runs both alembic commands + pytest and exits 0

## Do NOT touch
- Keycloak configuration
- Auth API endpoints (s1.3)
- Any other service tables
- `web/` or `mobile/`
