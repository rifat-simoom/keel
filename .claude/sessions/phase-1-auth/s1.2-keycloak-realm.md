# Session: Auth — Keycloak Realm Configuration
## Phase
1 — Auth & Identity

## Context files to load
- `SPEC.md` (section: Phase 1 — Auth Strategy, Keycloak Realm, Web Auth Flow, Mobile Auth Flow)
- `CLAUDE.md`
- `infrastructure/docker-compose.yml`

## Your task
Create `infrastructure/keycloak/realm-export.json` — a complete Keycloak realm export for the `keel` realm. It must define: three clients (`keel-web` public PKCE with redirect `https://keelapp.co.uk/callback` and `http://localhost:5173/callback`, `keel-mobile` public PKCE with redirect `com.keelapp://callback`, `keel-api` confidential for token introspection), three roles (`owner`, `accountant`, `employee`), and PKCE enforced on both public clients. Also create `infrastructure/keycloak/import-realm.sh` that imports the realm into a running Keycloak container using the admin CLI.

## Constraints
- PKCE must be enforced (not optional) on `keel-web` and `keel-mobile` — no implicit flow allowed
- `keel-api` is confidential and used only for server-side token introspection
- Both public clients must have PKCE method `S256`
- Realm name must be exactly `keel` (matches all env vars in SPEC.md)
- Do not store `KEYCLOAK_ADMIN_PASSWORD` in the realm export — it stays in docker-compose env
- Social login (Google, Apple) providers left as empty stubs — not configured in this session

## Done when
- `bash infrastructure/keycloak/import-realm.sh` runs against the Docker Keycloak container without error
- Keycloak admin UI at `http://localhost:8080` shows realm `keel` with all three clients and three roles
- `curl http://localhost:8080/realms/keel/.well-known/openid-configuration` returns 200 with JWKS URI
- `make verify` runs the curl health check and exits 0

## Do NOT touch
- Backend auth service code (s1.3)
- Web or mobile auth flows (s1.4, s1.5)
- Database schema (s1.1)
