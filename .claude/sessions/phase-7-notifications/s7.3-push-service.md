## Your task
Implement push notification dispatch in `backend/services/notifications/push.py` and the device registration endpoint: (1) `PushDispatcher` class with method `async send(notification: Notification, user_id: UUID) -> None` — looks up all `device_tokens` for the user, sends to Expo Push API (`https://exp.host/--/api/v2/push/send`) using the `exponent-server-sdk` Python package or raw `httpx`. Batch up to 100 tokens per Expo API call. Handle `DeviceNotRegistered` errors by deleting the stale token from `device_tokens`. (2) Add `POST /api/v1/notifications/register-device` endpoint — accepts `{ token: string, platform: "ios" | "android" }`, upserts into `device_tokens`, returns 200. (3) Add `DELETE /api/v1/notifications/register-device/{token}` — removes a device token on logout.

## Constraints
- Expo Push API batching: max 100 tokens per request — never exceed this
- `DeviceNotRegistered` ticket error: delete the token immediately from `device_tokens`
- Push payload must include `{ title, body, data: { route } }` — `route` enables deep linking
- `POST /register-device` is idempotent — `UNIQUE(user_id, token)` prevents duplicates (from s7.1)
- Never log the raw push token — log `platform` and first 8 chars only (FP-18)
- `send()` must not raise on Expo API failure — log error and continue (best-effort delivery)

## Done when
- `POST /register-device` stores token correctly, idempotent on repeat
- `PushDispatcher.send()` calls Expo API with correct payload format
- Stale token handling: `DeviceNotRegistered` response deletes the token
- `pytest backend/services/notifications/tests/test_push.py -v` passes (mock Expo API with `respx`)
- `make verify` runs pytest and exits 0

## Do NOT touch
- Email dispatch (s7.4)
- In-app notification API (s7.6)
- Mobile push setup (s7.7)
