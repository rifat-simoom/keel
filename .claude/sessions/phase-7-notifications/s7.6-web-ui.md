7 — Notifications & Deadlines

## Your task
Build the web in-app notification UI: (1) `web/src/components/notifications/NotificationBell.tsx` — bell icon in the sidebar top bar with unread count badge (red dot if >0). Polls `GET /api/v1/notifications` every 60 seconds via TanStack Query. (2) `web/src/components/notifications/NotificationDropdown.tsx` — dropdown panel listing the 10 most recent notifications, each with icon by type, title, relative timestamp ("2 minutes ago"), unread indicator. Clicking a notification: marks it read (`POST /api/v1/notifications/{id}/read`), navigates to `notification.route`. "Mark all read" button. (3) `web/src/components/notifications/DeadlineBanner.tsx` — full-width warning banner shown at top of every page when a deadline is within 7 days. Uses `GET /api/v1/deadlines/next`. Banner is dismissible per session (sessionStorage key). (4) Add `GET /api/v1/notifications`, `POST /api/v1/notifications/{id}/read`, `POST /api/v1/notifications/read-all` endpoints to the notifications router.

## Constraints
- Bell badge count shows only unread notifications (`read_at IS NULL`)
- Polling interval: 60 seconds — not shorter (avoid hammering the API)
- Deadline banner urgency: amber at 7d, red at 1d — not shown above 7 days
- Clicking notification navigates using React Router `useNavigate` — not a hard page reload
- "Mark all read" must be optimistic — immediately clears the badge, rolls back on API error
- Never show raw stack traces in the notification panel (FP-21)

## Done when
- Bell icon shows unread count that updates every 60s
- Clicking a notification marks it read and navigates to the route
- Deadline banner appears when next deadline is ≤7 days
- Banner is dismissible and stays dismissed for the session
- `pnpm --filter web test --run` passes notification bell and banner tests
- `make verify` runs vitest and exits 0

## Do NOT touch
- Mobile push setup (s7.7)
- Email dispatch (s7.4)
- Push notification infrastructure
