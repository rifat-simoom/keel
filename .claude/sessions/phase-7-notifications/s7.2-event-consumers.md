7 — Notifications & Deadlines

## Your task
Implement the RabbitMQ consumers in `backend/services/notifications/consumers.py`. For each event in the catalogue that targets the notification service, create a handler that: inserts a `notifications` row, then calls the dispatch layer (push + email — implemented in s7.3 and s7.4, stubbed here as `await dispatch_notification(notification)`). Consumers to implement: `invoice.created`, `invoice.sent`, `invoice.viewed`, `invoice.overdue`, `invoice.paid` (triggers `payment_received` type), `document.uploaded`, `document.matched`, `document.attention_required`, `transaction.created`, `tax.deadline_approaching` (published by tax service scheduler). Each handler must be idempotent — use `event.idempotency_key` to deduplicate. Wire all consumers into a single `backend/services/notifications/main.py` that starts the consumer process.

## Constraints
- Each consumer is idempotent — store processed `idempotency_key` in Redis with TTL 24h to deduplicate
- DB insert and dispatch call happen in the same logical unit — if dispatch fails, retry (not rollback)
- Consumer must scope the `user_id` lookup to the `company_id` from the event payload
- `notification.route` must be set correctly per type: `invoice_viewed` → `/invoices/{invoice_id}`, `payment_received` → `/transactions/{transaction_id}`, etc.
- All consumers use the same `notifications` DB session — single service, single DB
- Log each received event at INFO level with `event_type` and `company_id` — no amounts in logs (FP-18)

## Done when
- Publishing `invoice.overdue` to RabbitMQ creates a `notifications` row with `type='invoice_overdue'`
- Consuming the same event twice (same `idempotency_key`) creates only one notification row
- `notification.route` is correctly set for each event type
- `pytest backend/services/notifications/tests/test_consumers.py -v` passes with real RabbitMQ (testcontainers)
- `make verify` runs pytest and exits 0

## Do NOT touch
- Push dispatch implementation (s7.3)
- Email dispatch implementation (s7.4)
- Deadlines service (s7.5)
- Web or mobile UI
