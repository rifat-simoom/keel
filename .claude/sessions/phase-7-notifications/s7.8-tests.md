## Your task
Write the complete notifications test suite: (1) Consumer tests (testcontainers + real RabbitMQ): publish `invoice.overdue` → assert notification row created with correct type, title, route; publish same event twice with same `idempotency_key` → assert only one notification row. (2) Push tests (mock Expo API via `respx`): `send()` calls Expo API with correct token + payload; `DeviceNotRegistered` response deletes the token from DB; batch size respects the 100-token limit. (3) Email tests (mock Resend): subject line for `tax_deadline_1d` contains "tomorrow"; subject for `tax_deadline_30d` does not imply urgency; template renders the CTA button URL. (4) Deadlines tests: Group A company at 30d/7d/1d before VAT deadline → correct urgency; idempotency: Celery task run twice same day fires events once; cross-tenant: company A's deadlines don't include company B's invoice due dates. (5) In-app notification API tests: `GET /notifications` returns only current user's notifications (FP-3); `POST /{id}/read` sets `read_at`; `POST /read-all` clears all unread.

## Constraints
- testcontainers for all DB and RabbitMQ-dependent tests
- No real Expo API or Resend calls — mock with `respx` and `httpretty`/`responses`
- Consumer idempotency test: publish same event twice, assert DB has exactly one row
- Cross-tenant test: user B cannot read user A's notifications
- Coverage: 80%+ on consumers, push, email, and deadlines modules

## Done when
