## Phase
7 — Notifications & Deadlines

## Context files to load

## Your task
Create the Alembic migration and SQLAlchemy models for the notifications service: (1) `notifications` table — `id UUID PK`, `company_id UUID FK`, `user_id UUID FK`, `type TEXT NOT NULL` (enum: `invoice_viewed`, `invoice_overdue`, `payment_received`, `receipt_matched`, `tax_deadline_30d`, `tax_deadline_7d`, `tax_deadline_1d`, `payroll_due`), `title TEXT`, `body TEXT`, `route TEXT` (deep link target), `read_at TIMESTAMPTZ`, `created_at TIMESTAMPTZ`, `deleted_at TIMESTAMPTZ`. (2) `device_tokens` table — `id UUID PK`, `user_id UUID FK`, `token TEXT NOT NULL`, `platform TEXT NOT NULL` (CHECK: `ios` or `android`), `created_at TIMESTAMPTZ`, `last_seen_at TIMESTAMPTZ`, `UNIQUE(user_id, token)`. Add indexes on `notifications.user_id`, `notifications.read_at`, `notifications.created_at`.

## Constraints
- `notifications.type` use a PostgreSQL `CHECK` constraint with the full list of valid types
- `device_tokens` must enforce `UNIQUE(user_id, token)` — registering the same token twice is a no-op
- `notifications.deleted_at` present — soft delete only (FR-21)
- `notifications.route` stores the deep link path (e.g., `/invoices/abc-123`) — used for push payload
- Migration reversible

## Done when
- `alembic upgrade head` creates both tables cleanly
- `alembic downgrade -1` reverses cleanly
- `pytest backend/services/notifications/tests/test_models.py -v` passes (create notification, unique device token, type constraint, soft delete)
- `make verify` runs alembic + pytest and exits 0

## Do NOT touch
- Event consumers (s7.2)
- Push or email dispatch (s7.3, s7.4)
- Web or mobile UI
