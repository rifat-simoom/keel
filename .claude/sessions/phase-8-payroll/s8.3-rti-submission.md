## Your task
Implement RTI Full Payment Submission in `backend/services/payroll/rti.py`: (1) `RTISubmitter` class with `async submit_fps(payroll_run_id: UUID) -> RTIResult` — builds the HMRC FPS XML payload per the RTI schema (employee NI number, tax code, gross pay, PAYE deducted, NIC deducted), submits to HMRC Government Gateway (or sandbox endpoint during dev), handles response, stores the correlation ID, sets `payroll_runs.submitted_to_hmrc = True` and `payroll_runs.status = 'submitted'`. (2) Implement the HMRC Government Gateway sandbox endpoint: `https://test-api.service.hmrc.gov.uk/organisations/paye/{empref}/fps` — use the HMRC test credentials from env vars. (3) Publish `payroll.run` event via outbox after successful submission.

## Constraints
- FPS must be submitted **on or before** pay date — the API endpoint (s8.4) must enforce this (FR-19)
- Once `submitted_to_hmrc = True`, the payroll run is locked — no edits (FR-21)
- Audit log written on submission (FP-5)
- `payroll.run` event published via outbox (FP-8)
- HMRC credentials from env vars: `HMRC_SENDER_ID`, `HMRC_PASSWORD`, `HMRC_EMPLOYER_REF` — never hardcoded
- RTI submission is synchronous in Phase 8 (offload to Celery task in production hardening)
- On HMRC API error: do NOT set `submitted_to_hmrc = True` — return the error to the caller
- XML namespace and schema: follow HMRC RTI FPS schema v2023 structure

## Done when
- `submit_fps()` builds valid FPS XML with all required employee fields
- Successful sandbox submission sets `submitted_to_hmrc = True` and stores correlation ID
- HMRC API error leaves `submitted_to_hmrc = False` and returns the error
- Second call on already-submitted run raises `PayrollAlreadySubmittedError`
- `payroll.run` event published after successful submission
- `pytest backend/services/payroll/tests/test_rti.py -v` passes (mock HMRC API with `respx`)
- `make verify` runs pytest and exits 0

## Do NOT touch
- PAYE calculator (s8.2)
- API endpoints (s8.4)
- Web UI
