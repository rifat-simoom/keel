## Your task
Implement transaction-document matching in `backend/services/documents/matching.py` and expose it via two endpoints: (1) `POST /api/v1/documents/{id}/match` — links a document to a transaction: sets `document.transaction_id`, sets `transaction.receipt_id`, sets `document.status = 'matched'`, writes audit log on both records, publishes `document.matched` event. (2) `DELETE /api/v1/documents/{id}/match` — unlinks: clears both FKs, resets `document.status = 'uploaded'`, writes audit log. (3) `GET /api/v1/reconciliation/unmatched` — returns two lists: unmatched debits (transactions with no `receipt_id`) and unmatched credits (transactions with no `invoice_id`). This powers the reconciliation dashboard counter.

## Constraints
- Matching must verify document and transaction belong to the same `company_id` — cross-tenant match is forbidden (FP-3)
- Both DB records updated in a single transaction — atomic (if one fails, both roll back)
- Audit log written on **both** the document and the transaction records (FP-5)
- `document.matched` event published via outbox (FP-8)
- Unmatched count endpoint: scopes to `current_user.company_id` only (FP-3)
- Cannot match a document that already has `deleted_at` set (FP-16)

## Done when
- `POST /match` links document to transaction and both FKs are set
- `POST /match` with document from company A and transaction from company B returns 404
- `DELETE /match` unlinks and resets both records
- `GET /reconciliation/unmatched` returns correct counts
- `document.matched` event is published to RabbitMQ
- `pytest backend/services/documents/tests/test_matching.py -v` passes
- `make verify` runs pytest and exits 0
