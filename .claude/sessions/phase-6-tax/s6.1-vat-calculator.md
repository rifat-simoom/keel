## Your task
Implement `backend/services/tax/vat_calculator.py` — the `VATCalculator` class with method `async calculate_return(company_id, period_start, period_end) -> VATReturn`. The method must: (1) Look up the company's `vat_scheme` from `companies` table. (2) For `accrual` scheme: sum output VAT from invoices with `issue_date` in the period and `status IN ('sent', 'viewed', 'paid', 'overdue')`. (3) For `cash` scheme: sum output VAT from invoices with `paid_at` in the period and `status = 'paid'`. (4) Sum input VAT from `documents` with `document_date` in the period, `status = 'matched'`, category NOT in (`ENTERTAINMENT`, `WAGES`, `BANK_CHARGES`, `INSURANCE`). (5) Return a `VATReturn` dataclass with `box_1` through `box_7`, `period_start`, `period_end`, `company_id`, `net_vat_payable`. Also create the `VATReturn` SQLAlchemy model in `backend/services/tax/models.py` with `submitted_at TIMESTAMPTZ` (null until submitted) and `submission_reference TEXT`.

## Constraints
- All arithmetic uses `Decimal` — never `float` (FP-2)
- VAT rates imported from `shared.constants.tax` — never hardcoded (FP-9)
- `ENTERTAINMENT` expenses never included in input VAT — they are not reclaimable (FR-2, FP-13)
- Scheme logic (cash vs accrual) must branch on `company.vat_scheme` from DB (FR-5)
- `VATReturn` model has `deleted_at` and `submitted_at` — once `submitted_at` is set, the record is immutable (FR-21)
- `box_5` = `box_1 - box_4` — if positive, company owes HMRC; if negative, HMRC owes company
- Box calculations must include the `currency` breakdown (all GBP in Phase 6)

## Done when
- `VATCalculator.calculate_return()` returns correct `box_1`–`box_7` for a company with 3 paid invoices and 2 matched expenses
- Cash scheme: only paid invoices included; accrual scheme: sent invoices included
- `ENTERTAINMENT` expenses never appear in `box_4`
- `pytest backend/services/tax/tests/test_vat_calculator.py -v` covers both schemes, ENTERTAINMENT exclusion, and empty-period edge case
- `make verify` runs pytest and exits 0
