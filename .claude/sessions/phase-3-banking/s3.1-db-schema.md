# Session: Banking — Database Schema & Models
## Phase
3 — Banking & Transactions

## Context files to load
- `SPEC.md` (section: Phase 3 — Database)
- `CLAUDE.md`
- `.claude/guardrails/financial-rules.md` (sections FR-1, FR-2)
- `backend/services/auth/models.py`
- `backend/shared/models/audit.py`

## Your task
Create the Alembic migration and SQLAlchemy 2.0 async models for `accounts`, `transactions`, and `virtual_cards` tables exactly as specified in SPEC.md Phase 3. Extend the migration to also add `invoice_id` FK on `transactions` (references `invoices(id)` — nullable, for Phase 4 reconciliation). Add `deleted_at`, `created_by`, `updated_by` to all three tables. Add indexes on `transactions.account_id`, `transactions.transaction_date`, `transactions.category`, `transactions.is_expense`.

## Constraints
- `transactions.amount` must be `NUMERIC(12,2)` — positive for credits, negative for debits (FR-1)
- `transactions.currency` column required — default `'GBP'` (FR-24, FP-20)
- `accounts.balance` is `NUMERIC(12,2)` — never `FLOAT` (FP-2)
- `virtual_cards.status` constrained to: `active`, `frozen`, `cancelled` — use `CHECK` constraint
- All three tables get `company_id UUID NOT NULL REFERENCES companies(id)` (CLAUDE.md rule 3)
- FK `transactions.invoice_id` is nullable (reconciliation happens in Phase 4)
- Migration must be reversible

## Done when
- `alembic upgrade head` applies all three tables cleanly
- `alembic downgrade -1` reverses cleanly
- `pytest backend/services/banking/tests/test_models.py -v` passes (create account, create transaction, create virtual card, FK integrity, CHECK constraint on card status)
- `make verify` runs alembic + pytest and exits 0

## Do NOT touch
- API endpoints (s3.2)
- Web or mobile UI
- `invoices` table or invoice service (Phase 4)
- `receipt_id` column on transactions (Phase 5)
