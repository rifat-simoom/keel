## Your task
Implement all 8 banking API endpoints in `backend/services/banking/router.py`: `GET /api/v1/accounts/me`, `GET /api/v1/transactions` (paginated, filterable by `date_from`, `date_to`, `category`, `min_amount`, `max_amount`), `GET /api/v1/transactions/{id}`, `POST /api/v1/transactions` (manual top-up / test data entry), `GET /api/v1/accounts/me/card`, `POST /api/v1/accounts/me/card/freeze`, `POST /api/v1/accounts/me/card/unfreeze`, `GET /api/v1/accounts/me/stats` (income total, expense total, net balance over a date range). All endpoints require auth. `POST /transactions` publishes `transaction.created` event via outbox.

## Constraints
- All endpoints scope queries to `current_user.company_id` — never global queries (FP-3)
- `POST /transactions` requires `Idempotency-Key` header (FP-14)
- Pagination: `page` + `page_size` query params, response includes `total`, `page`, `page_size`, `items`
- `transactions.amount` stored as signed `Decimal` — positive = credit, negative = debit (FR-1)
- Freeze/unfreeze must write audit log entry (FP-5)
- `GET /accounts/me/stats` never uses float arithmetic — all `Decimal` aggregations (FP-2)
- `transaction.created` event published via outbox pattern (FP-8)
- Card `cvv` field must never be returned in any API response — last_four only

## Done when
- `GET /accounts/me` returns account with balance, sort_code, account_number
- `GET /transactions` returns paginated list filterable by all params
- `POST /transactions` creates transaction and publishes event, idempotent on repeat key
- `POST /card/freeze` changes card status and writes audit log
- `pytest backend/services/banking/tests/test_router.py -v` passes all endpoint tests
- `make verify` runs pytest and exits 0

## Do NOT touch
- Web or mobile UI (s3.3, s3.4)
- `invoice_id` reconciliation logic (Phase 4)
- Virtual card creation (card is seeded, not created via API in Phase 3)
